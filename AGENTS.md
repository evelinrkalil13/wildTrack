# WildTrack Platform - Project Context for Codex

## Project Summary

WildTrack is an IoT-based wildlife monitoring platform for intelligent feeding stations.

The system will allow users to register, manage feeding stations, associate stations with geographic zones, register animals identified by RFID, configure food types, receive IoT events from ESP32-based feeders, and visualize station activity through a web platform and a geoportal.

The project must be developed using Spec-Driven Development. Do not generate implementation code before the specification, architecture, data model, API contracts, and task breakdown are completed.

## Main Goal

Build a clean, maintainable, modular web platform composed of:

1. Administration platform
2. Geoportal / geovisor
3. IoT event ingestion backend
4. Analytics dashboard

The MVP must be simple enough to build, but structured enough to evolve.

## Chosen Architecture

Use a modular monolith.

Do not create microservices for the MVP.

The backend must be a single FastAPI application organized by business modules.

Each module should have its own:

* router
* schemas
* models
* repository
* service
* exceptions when needed

The backend must avoid mixing business logic inside route handlers.

## Important Development Rules

* The IoT device must never write directly to PostgreSQL or MongoDB.
* The IoT device sends events through MQTT.
* The backend receives, validates, processes, and stores IoT events.
* PostgreSQL stores structured/master data.
* MongoDB stores IoT events, telemetry, visit records, alerts, and flexible event payloads.
* Multimedia files such as photos or videos should not be stored directly in PostgreSQL or MongoDB. Store only their URLs or metadata.
* The frontend must only communicate with the backend API.
* The geoportal must not read directly from databases.
* Use clean architecture principles without overengineering.
* Keep the MVP realistic and buildable.

## Core Business Concepts

### User

A user can register in the platform and can be associated with one or more feeding stations.

Users may have roles such as:

* admin
* researcher
* field_operator

### Station

A station represents an intelligent feeder.

A station belongs to one geographic zone.

A station can have many associated users.

A station can have many associated animals.

A station can have one active food configuration.

### Zone

A zone stores geographic and environmental context.

Fields may include:

* id
* name
* municipality
* city
* country
* altitude
* latitude
* longitude

A zone can have multiple stations.

### Animal

An animal may be identified by RFID.

Fields may include:

* id
* station_id
* rfid_tag
* species
* sex
* estimated_age
* is_identified

### Food

Food represents the type of food used by a station.

Fields may include:

* id
* name
* type
* description

### IoT Event

An IoT event is generated by a feeding station.

Events should be stored in MongoDB.

Expected event data:

* event_id
* event_type
* station_id
* animal_id, if applicable
* rfid_tag, if applicable
* timestamp
* temperature
* humidity
* consumed_grams
* initial_weight
* final_weight
* latitude
* longitude
* media_url, if applicable
* device_status
* raw_payload

## Main IoT Event Flow

1. Animal approaches the feeder.
2. Presence sensor detects activity.
3. RFID reader attempts to identify the animal.
4. Camera may capture photo or video.
5. Temperature and humidity are measured.
6. Load cell measures initial and final food weight.
7. ESP32 calculates estimated food consumption.
8. ESP32 publishes event through MQTT.
9. Backend receives event.
10. Backend validates station existence.
11. Backend links RFID tag to animal when possible.
12. Backend stores event in MongoDB.
13. Backend may generate alerts.
14. Geoportal and dashboard consume processed data through API endpoints.

## Main Platform Flow

1. User registers.
2. User logs in.
3. User registers a station.
4. Browser may request geolocation permission.
5. Station location is stored in PostgreSQL/PostGIS.
6. User associates food type.
7. User registers animals and RFID tags if available.
8. User monitors station activity from dashboard.
9. User visualizes stations and events from geoportal.

## Geoportal Requirements

The geoportal must show:

* station locations
* station status
* recent events
* heatmap of activity
* heatmap of consumption
* temperature and humidity by location
* events by zone
* station popup with summary data

## Dashboard Requirements

The dashboard must show:

* total stations
* active stations
* disconnected stations
* total events
* identified animals
* unidentified events
* consumption over time
* visits over time
* consumption by zone
* events by station
* environmental comparison charts

## MVP Scope

The first version must include:

1. User registration and login
2. Station registration
3. Zone registration
4. User-station association
5. Animal registration with RFID tag
6. Food registration
7. MQTT event ingestion
8. MongoDB event storage
9. Basic REST API for dashboard
10. Basic REST API for geoportal
11. React frontend with admin views
12. Leaflet geoportal
13. Docker Compose environment

## Out of Scope for MVP

Do not implement in the first version:

* machine learning
* image recognition
* mobile native app
* microservices
* Kubernetes
* advanced role permissions
* payment systems
* real-time WebSocket dashboard unless simple
* production cloud deployment automation

## Spec-Driven Development Process

Follow this order strictly:

1. Requirements specification
2. Architecture document
3. Data model document
4. API contract document
5. IoT event contract document
6. Task breakdown
7. Backend scaffold
8. Frontend scaffold
9. Infrastructure setup
10. Module-by-module implementation

Do not start implementation until the specification documents are approved.
