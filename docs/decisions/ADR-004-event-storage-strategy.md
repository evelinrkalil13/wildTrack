# ADR-004 — Event and Multimedia Storage Strategy

**Status:** Accepted  
**Date:** 2026-06-13  
**Deciders:** WildTrack platform team  
**Related documents:** SDD-01 v1.1.0, ADR-002-database-strategy.md

---

## Context

WildTrack must store three distinct categories of data generated by IoT field devices:

1. **Structured feeding events and sensor readings** — high-frequency, schema-flexible documents produced by ESP32 feeders on each animal visit.
2. **Device telemetry heartbeats** — periodic low-overhead status messages carrying firmware version, connectivity signal, memory usage, and uptime.
3. **Multimedia files** — photos and videos optionally captured by a camera module during a feeding session.

These categories have materially different characteristics in terms of volume, schema stability, query patterns, and size. A single storage system cannot serve all three well. This ADR defines which technology handles each category and why.

---

## Decision Drivers

- IoT events arrive with flexible, evolving payloads; a rigid relational schema would require frequent migrations as device firmware evolves.
- Multimedia files can be tens of megabytes each and must never be embedded in a database row or document.
- Device telemetry heartbeats arrive frequently (every 30–60 seconds per device) and need fast time-series retrieval without impacting transactional master data.
- The system must remain deployable locally via Docker Compose without requiring managed cloud services.
- Object storage access must be controlled; files must not be publicly browsable.

---

## Decision

### IoT Events → MongoDB (`iot_events` collection)

IoT feeding events are stored as documents in MongoDB.

**Rationale:**

- Event payloads vary between firmware versions and device types. A document model accommodates new fields without schema migrations.
- MongoDB's aggregation pipeline efficiently supports the time-series grouping, zone-level rollups, and consumption summaries required by the analytics and geoportal modules.
- Keeping events in MongoDB isolates high-write IoT traffic from the PostgreSQL instance that manages structured master data.
- The `raw_payload` field on each document preserves the original MQTT message for debugging and audit purposes.

**Collection:** `iot_events`  
**Key indexed fields:** `station_id`, `device_id`, `animal_id`, `rfid_tag`, `timestamp`, `event_type`

### Device Telemetry → MongoDB (`device_telemetry` collection)

Periodic device heartbeats are stored as documents in a dedicated MongoDB collection, separate from feeding events.

**Rationale:**

- Telemetry messages are structurally different from feeding events (no animal, no food consumption) and should not be mixed into the same collection.
- A dedicated collection allows independent indexing strategies optimized for time-series access by `device_id` and `timestamp`.
- Separating telemetry from events simplifies the analytics layer: feeding event queries are never polluted by heartbeat records.
- Future time-series indexes (MongoDB 5+ time series collections) can be applied to this collection without affecting event storage.

**Collection:** `device_telemetry`  
**Key indexed fields:** `device_id`, `station_id`, `timestamp`  
**Mandatory heartbeat fields:**

| Field | Type | Description |
|-------|------|-------------|
| `telemetry_id` | string (UUID) | Unique identifier for this record |
| `device_id` | string | FK to PostgreSQL `devices.id` |
| `station_id` | string (nullable) | FK to PostgreSQL `stations.id` at time of reading |
| `timestamp` | datetime (UTC) | Time the heartbeat was generated on the device |
| `firmware_version` | string | Semantic version string (e.g., `1.2.3`) |
| `wifi_signal` | integer (dBm) | WiFi RSSI at time of heartbeat |
| `uptime` | integer (seconds) | Seconds since last device reboot |
| `free_memory` | integer (bytes) | Available heap memory on the ESP32 |
| `device_status` | string | Self-reported device status (`ok`, `warning`, `error`) |
| `battery_level` | integer (%, nullable) | Battery percentage; null if device is wired (reserved for future use) |
| `raw_payload` | object | Original MQTT message payload preserved verbatim |

### Multimedia Files → MinIO Object Storage

Binary media files (photos, videos) are stored in a MinIO instance. MongoDB stores only metadata and object references; no binary data enters any database.

**Rationale:**

- Storing large binary files in MongoDB documents would bloat the database, exceed BSON document size limits (16 MB), degrade query performance, and complicate backup and restore procedures.
- MinIO provides an S3-compatible API that is self-hosted, deployable in Docker Compose, and behaves identically to AWS S3 — making a future migration to managed object storage straightforward.
- Pre-signed URLs allow the frontend to retrieve media directly from MinIO without routing binary payloads through the FastAPI backend, reducing bandwidth and latency.
- Access control is enforced at the bucket level; the bucket is private and all access is mediated by the backend or time-limited pre-signed URLs.

**MinIO bucket:** `wildtrack-media` (private)  
**Object key pattern:** `{station_id}/{year}/{month}/{device_id}_{timestamp_iso}_{original_filename}`  
**Supported MIME types (MVP):** `image/jpeg`, `image/png`, `video/mp4`, `video/x-msvideo`  
**Maximum file size (MVP):** 10 MB  
**Pre-signed URL expiry:** 15 minutes (configurable)

**Media metadata stored in MongoDB (`media_metadata` collection):**

| Field | Type | Description |
|-------|------|-------------|
| `media_id` | string (UUID) | Unique identifier |
| `event_id` | string | Reference to the associated `iot_events` document |
| `station_id` | string | Station that generated the media |
| `device_id` | string | Device that captured the media |
| `media_type` | string | `photo` or `video` |
| `object_key` | string | MinIO object key (bucket-relative path) |
| `url` | string | Full object URL (internal MinIO endpoint) |
| `file_size_bytes` | integer | File size in bytes |
| `captured_at` | datetime (UTC) | Timestamp from device at time of capture |

## Alternatives Considered

### Store all events in PostgreSQL

**Rejected.** PostgreSQL is the right choice for relational master data but poorly suited for high-frequency, schema-flexible IoT event documents. Adding a JSONB column to absorb event payloads would work at small scale but complicates aggregation queries and mixes concerns across the schema boundary established in ADR-002.

### Store telemetry in the same collection as feeding events (`iot_events`)

**Rejected.** Mixing heartbeat records and feeding events in a single collection forces every feeding event query to filter out telemetry by event type, adds noise to the analytics pipeline, and prevents independent indexing and retention strategies per data type.

### Use GridFS for multimedia storage

**Rejected.** GridFS stores binary content inside MongoDB using a chunked file system abstraction. This adds complexity to reads and writes, prevents efficient streaming, does not support CDN integration, and contradicts the principle of keeping MongoDB reserved for structured document data.

### Use an external managed object store (AWS S3, GCP GCS)

**Deferred (post-MVP).** Managed object storage is the long-term target. For the MVP, MinIO in Docker Compose provides an identical API surface and allows development without cloud accounts or network dependencies. Migration requires only changing the endpoint URL and credentials in environment variables.

### Store media files on the local filesystem

**Rejected.** Local filesystem storage does not compose with Docker containers, has no access control primitives, and cannot scale beyond a single host. MinIO solves all three problems with no additional API complexity.

---

## Consequences

### Positive

- Each storage system is optimized for its data type: PostgreSQL for relational master data, MongoDB for flexible document events and telemetry, MinIO for binary objects.
- Schema changes to IoT event or telemetry payloads do not require database migrations.
- MinIO-to-S3 migration in production is a configuration change, not a code change.
- Pre-signed URLs offload media bandwidth from the FastAPI backend.
- Dedicated `device_telemetry` collection enables independent monitoring, alerting, and future time-series optimizations.

### Negative / Trade-offs

- Three distinct storage systems increase infrastructure complexity and the number of services in Docker Compose.
- Cross-collection queries (e.g., joining event metadata with media metadata) must be handled in the application service layer, as MongoDB does not support cross-collection joins natively.
- Dead-letter event review requires MongoDB tooling (Compass, Atlas, or a custom admin view) rather than standard SQL queries.

### Neutral

- The `battery_level` field in device telemetry is reserved but will remain null for wired devices in the MVP. It is included in the schema now to avoid a future breaking change in the MQTT contract.
- Pre-signed URL expiry of 15 minutes is a balance between security and user experience. This should be reviewed if the geoportal needs to display media in long-lived sessions.
